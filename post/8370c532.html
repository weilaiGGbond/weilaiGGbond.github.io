前端面经-前端React技术栈

发表于2025-03-27|更新于2025-05-26
|浏览量:

react 中常用的 hook

useState:用于在函数组件中声明和管理状态,它返回的是一个数组，其中包括当前状态值和更新状态的函数。

useEffect:用于处理函数组件中的副作用，常用于处理数据获取、订阅事件、定时器函数等。useEffect接收两个参数，一个是函数，一个是用来控制副作用函数执行时机依赖数组。如果没有传递数组参数就默认在每次组件渲染后执行，如果传递了数组参数，就会根据数组中参数发生变化后执行。

useContext:用于访问 react 上下文的值。上下文允许数据在不同组件间共享，而不需要通过组件逐层向下传递 props

useCallback:用于记忆组件方法，避免在每次渲染时都要重新创建函数，从而提高性能。useCallback 接收两个参数，分别是记忆化的回调函数 和 依赖项的数组 。主要的应用场景有避免子组件不必要的渲染，和保存回调函数的状态

useMemo:用户缓存计算结果，避免在每次渲染时重新计算。useMemo 接收两个参数，分别是 执行的计算逻辑函数 和 依赖项数组 。useMemo 常见的应用场景有性能优化，避免不必要的引用变化

useReducer:用于处理复杂状态逻辑或多个子状态的更新，useReducer接收两个参数，一个reducer函数和初始状态，并返回当前状态和一个 dispatch 函数。与 useState 不同，useReducer 使用 reducer 来决定如何更新状态，这使得它在处理复杂状态或多个动作时更加清晰和可维护

useLayoutEffect:用法和useEffect类似，但它的执行时机有所不同。它在所有DOM 更新和渲染完成后，同步执行，以便在绘制到屏幕之前对 DOM 进行操作

useMemo 和 useCallback 之间的区别

useMemo 用于对复杂计算结果进行记忆化，它接收的是创建函数和依赖项数组，当依赖项中的数据发生变化时，这个创建函数才会被执行。而对于useCallback 用于对函数本身进行记忆化，常用于阻止子组件不必要的渲染，接收内联函数和依赖项数组，但是与useMemo不同的是useCallback在依赖项发生变化时会重新创建内联函数，而且useMemo返回的是创建函数的返回值，useCallback返回的是记忆化的函数本身

useEffect 是如何判断依赖改变的？

useEffect 使用浅比较来判断依赖项是否发生变化。意味着只比较对象和数组的引用，而不深入比较对象或数组内部的属性或元素

useState 为什么返回数组而不是对象？

如果使用对象对返回的数据进行获取会导致命名重复,需要对变量进行重新命名。但使用数组就可以在接收数据时进行重命名，减少了变量命名冲突。避免解构时的命名冲突、灵活的命名避免记忆负担、遵循 Hooks 设计的间接性

useState 和 useReducer 之间的区别

useReducer 的更新逻辑集中在 reducer 函数，通过不同 action 类型处理，使更新逻辑清晰、可预测。useState更新逻辑在更新函数中实现，而且每次调用useState都会触发组件的重新渲染，useReducer 可以通过reducer函数来比较新旧状态优化性能，避免不必要的渲染

useContext 的优缺点

useContext 提供了一个组件树中的数据共享机制，不需要组件之间通过props进行层层传递，但是useContext会导致调试和组件的复用变得困难，当组件中使用了context，则该组件只能作为创造这个context父组件下的组件使用

useLayoutEffect 和 useEffect之间的区别

useLayoutEffect会在DOM 更新后、浏览器绘制前执行，由于useLayoutEffect是同步任务，会阻塞浏览器绘制，而useEffect会在浏览器绘制之后异步执行，useEffect不会阻塞浏览器渲染，能够提高渲染性能。

React 组件通信的方式

父子组件间 通信：可以通过props实现。父组件可以将数据或函数作为props传递给子组件，子组件可以通过调用父组件传递的回调参数并传递数据

兄弟组件间通信：可以通过父组件作为中间件，将子组件通过回调函数将数据传递给父组件，再由父组件将数据传递给另一个兄弟组件

跨层级组件通信：可以使用React中的Context API实现一个组件树中的数据共享，而不必使用 props 层层传递

事件总线 Event Bus：事件总线是一种全局的事件系统，组件可以在事件总线上发布事件和订阅事件，从而实现组件间的通信。不过，在 React 中要谨慎使用事件总线，它会增加代码的耦合度

Redux：使用redux全局状态管理工具来管理组件间通信和共享状态，redux提供了全局的状态容器，组件可以从中获取数据和触发状态更新。

受控组件和非受控组件的区别

受控组件和非受控组件是 React 中处理表单元素的两种不同方式

受控组件：表单元素的值有 React 的 state 完全控制，并通过 onChange 事件同步更新

非受控组件：表单元素的值由 DOM 自身管理，通过 ref 在需要时获取输入值，非受控组件常用于文件上传、富文本编辑

React 18 有什么更新？

新特性：

ReactDOM.createRoot代替render，启动 React 18 中的 concurrency并发渲染

setState 自动批处理: 对于多次状态更新会自动批处理，这意味着多个更新会在同一个渲染周期内合并，从而提高性能

新 API

useDeferredValue用于延迟渲染不紧急的部分，将数据进入慢速渲染通道中，与防抖类似但没有固定的延迟时间，常用于根据用户输入渲染结果。

useTransition用于改进用户体验，特别是在执行可能导致页面延迟的操作

prop 和 state 的区别

prop是由父组件传递的，默认下不可直接修改，state是组件内部的数据，属于自身私有状态，是可以通过 setState 来进行修改的。二者进行数据性能优化的方式也不同，prop常使用 React.memo 来避免子组件的渲染，state常使用useCallback来进行依赖优化来避免重新渲染

Hooks 解决了什么问题？

更好的逻辑组织和代码分离：Hooks 允许开发者按照逻辑关系而非生命周期方法来组织代码，使得同一逻辑下的代码更加集中，提高了代码的可读性和可维护性

在函数组件中使用状态和其他 React 特性：扩展了函数组件的功能

React Hook 的使用限制有哪些

限制有：

只能在函数组件的最顶层调用 Hooks，不能在循环、条件语句或者嵌套函数中调用 Hooks，是为了确保 Hooks 在每次组件渲染时都以相同的顺序调用，便于 React 追踪依赖

只能在 React 函数组件或自定义 Hooks 中调用 Hooks，不能再普通的 JS 函数中调用 Hooks，是为了确保只在 React 的上下文中使用 Hooks

限制的原因：Hooks 的规则是为了让 React 正确的追踪状态和副作用，同时保持代码的可预测性。以牺牲代码的灵活性为代价，换取更简洁、可维护的状态管理方案

类式组件和函数组件的异同

类式组件是基于面向对象编程的，它主打的是继承、生命周期等核心概念，而函数组件内核时函数式编程，更多强调的是没有副作用，引用透明等特点。在性能优化上，类组件主要依靠 shouldComponentUpdate 阻断渲染提升性能，而函数组件依靠 React.Memo渲染结果来提升性能，而且类式组件在并发模式中由于生命周期带来的复杂度，并不易于优化，函数组件更加轻量化，在 Hooks 的基础上提供了更加细粒度的逻辑组织和复用

JSX 的本质

JSX 是语法糖，在被Babel编译成React.creatElement()函数调用，最终生成React元素

在 Fiber 架构下，JSX转换成真实DOM的过程

JSX 转换成虚拟 DOM JSX会被Babel遍历成React.creatElement函数转换成虚拟 DOM 对象

创建 Fiber 节点 React 会将虚拟 DOM 对象映射为 Fiber 节点。每个 Fiber 节点包含关于组件的信息，如类型、属性、状态以及指向父子节点的引用

调和阶段 在调和阶段，React 会比较当前 Fiber 树和新的 Fiber 树之间的差异，以确定哪些节点需要更新、添加或者删除

提交阶段 在提交阶段，React 会根据调和阶段的结果将变更应用到真实的 DOM 中

高阶组件和普通组件的区别

高阶组件是 React 中的一种逻辑复用模式，通过函数式编程的组合思想，将组件间共享的逻辑抽象出来，而不是通过继承或者直接修改组件。常用于代码复用和逻辑抽象，还有条件渲染和拦截。

React 中如何是实现响应式的

React 实现响应式的核心在于状态管理和虚拟 DOM 技术。当组件的状态state或属性prop发生变化时就会触发组件重新渲染。组件内部状态或从父组件传入的属性发生变化时，React 会调用更新函数，标记组件需要更新，然后React根据最新的状态构建新的虚拟DOM树，通过diff算法找出patch，通过比较结果对实际 DOM 进行最小量、必要的修改，借助Fiber架构的调度机制，更合理的分配更新任务，优化更新优先级，确保了用户界面的平滑和高效响应

Redux

Redux 是一个基于 单向数据流 和 不可变状态 的 React 状态管理库

redux 的工作流程：

触发 Action : 用户操作或事件触发Action

Dispatch Action: 通过 store.dispatch(action)将Action发送到Store

Reducer 处理 ：Store 调用Reducer，传入当前状态和Action，生成新状态

更新Store:Store更新为新状态

通知订阅者：所有订阅了Store的组件会收到通知并重现渲染

Redux Toolkit 是为解决Redux的冗长问题，它通过使用createSlice自动生成Action和Reducer来简化Reducer和通过createAsyncThunk简化异步逻辑

react-router

主要的路由模式：

HashRouter: 利用 URL 中的哈希(#)部分来模拟不同的路径,不会触发页面更新

BrowserRouter: 利用 HTML5 的 History API 来管理浏览器历史记录和 URL 路径，需要服务器支持，确保刷新页面时能够正确处理路由

MemoryRouter: 将路由历史记录存储在内存中，不依赖于浏览器的地址栏，适合需要在内存中管理路由历史的场景

StaticRouter: 用于服务端渲染，特别是没有window环境下的NodeJS环境中，StaticRouter只关心当前匹配的路由内容，不关心其他路由状态，性能较高

Router 的路由跳转和参数传递

路由跳转：

使用 useNavigate Hook 来实现

使用 Navigate 组件 <Navigate to="/target-path" replace />;

参数传递：

路径参数：

在路由路径中定义动态部分，这些部分会作为参数传递给目标组件

形式：/search/:id 获取方式：useParams

查询参数：

查询参数是在URL的问号后添加的键值对，用于传递额外的数据

形式：/search?id=1234 获取方式：useSearchParams

状态参数：

状态参数允许在导航时传递一些数据，这些数据不会显示在 URL 中

形式：navigate('/path',{state:data}) 获取方式：useLocation

Fiber 架构

Fiber 是 React 中为解决大型应用中的性能瓶颈，并支持异步渲染和优先级调度等高级特性，核心目标是通过可中断的增量渲染优化用户体验

Fiber 架构将渲染任务拆分为可中断的”小任务”，通过增量渲染在浏览器空闲时段分片执行，同时引入优先级调度，根据任务类型动态调整执行顺序，支持并发渲染，提高性能，解决了 Stack Reconclier中的阻塞用户交互和缺乏优先级控制以及不可中断的递归遍历等问题。

Fiber 架构的核心原理 是通过链表结构的 Fiber 节点将组件树转换成可中断、可恢复的异步任务单元，利用双缓冲机制，也就是通过Current Fiber 树和WorkInProgress Fiber 树来实现交替更新和优先调度模型，将渲染的过程拆分成可时间切片的增量任务，在浏览器空闲期分阶段执行，先协调计算差异，再同步提交 DOM 更新，从而实现高优先级任务的即时响应，避免主线程阻塞，提升复杂的渲染性能和流程的交互体验

Fiber 的遍历方法 通过迭代式链表结构（child、sibling、return 指针）模拟传统深度优先遍历（DFS），但核心区别在于其可中断性和任务分片：传统 DFS 通过递归一次性完成整个树遍历，无法暂停；而 Fiber 将遍历拆解为独立处理每个节点的增量任务，利用浏览器空闲时间片动态调度，允许高优先级任务中断并抢占当前遍历，随后从中断点恢复，从而避免主线程阻塞，实现异步渲染与响应式交互。

React Fiber 的优先级调度 基于 Lane 模型，通过二进制位掩码将任务划分为不同优先级车道，利用位运算高效合并或分离任务；调度器根据浏览器空闲时间片（requestIdleCallback）动态分配任务执行顺序，高优先级任务可抢占低优先级任务，通过时间切片拆分长任务，确保主线程及时响应用户操作，同时渐进完成低优任务，最终实现流畅的异步渲染与交互体验。

Fiber 架构下 React 中从状态更改到 DOM 更新有哪些同步或者异步操作

首先进行的是调用useState的回调函数，该过程是同步进行的，能够立即执行将更新的请求加入到React的更新队列中；然后进入状态更新的阶段，React默认会进行自动批处理，即对同一事件循环中的更新进行合并，以减少不必要的渲染（可以调用flushSync跳过自动批处理，让更新立即生效），该阶段进行的异步操作，在自动批处理完成后进入优先级调度阶段，React会通过Scheduler模块对任务分配优先级。调度的本质是异步的，高优先级任务可中断低优先级任务的执行，优先处理自身，待完成后恢复被中断的低优先级任务，以保证高优先级任务的响应速度，然后是协调阶段，该阶段会异步创建新的Fiber树，通过时间分片将任务拆分为多个可中断的Fiber节点，并比较新旧树差异生成补丁，最后进入提交阶段，将补丁应用到真实 DOM 中，该阶段必须是同步且不可中断的，要保证 DOM 一次性更新到最新状态，避免中间状态闪烁，提交阶段后，useLayoutEffect 的回调会同步执行（在 DOM 更新后，浏览器渲染前执行，可能会阻塞浏览器渲染过程），而 useEffect 的回调会被异步调度到微任务队列（在浏览器渲染后执行）

React 中的事件捕获和事件冒泡的实现原理

React 的事件系统是在原生的 DOM 事件机制上做了一层封装，实现了一套合成事件系统，以达到在不同的浏览器之间事件处理差异的目的，在 React 中不会将事件绑定到具体的 DOM 元素上，而是将所有事件统一绑定到根节点的容器上。首先在用户点击之前 React 在组件挂载时通过调用 SimpleEventPlugin 插件函数来注册事件，将原生事件名称处理为 React 事件名称，将原生事件名称与 React 事件名称建立 Map 映射关系，然后将收集的原生事件名称加入到 allNativeEvents 数组中，为其映射出 React 中的捕获和冒泡名称，然后在调用 creatRoot 函数时调用内部的 listenToAllSupportedEvents 函数，并创建 FiberRoot 以监听所有的原生事件，根据之前获取的 allNativeEvents 数组对其进行遍历，然后通过 listenToNativeEvent 创建 listener 监听函数，listener 函数会通过 dispatchEvent 将 this 绑定到事件名称、触发容器以及事件类型中，并返回一个函数，最后通过调用 addEventBubbleListener addEventCaptureListener 函数，在 root 根容器中绑定原生事件的冒泡和捕获事件
