前端面经-前端工程化

性能优化

前端可以做的性能优化有哪些？

使用 CDN 进行静态资源的缓存加速资源的加载速度

对图片/组件进行懒加载，为图片添加上loading="lazy",当图片滚动到可视区域时再加载，对于组件懒加载来说可以使用 import()+箭头函数来实现按需导入

使用防抖和节流来减少对服务器/资源的请求，防抖是再延迟期间内只执行最后一次，节流是在规定时间内只执行一次

减少重绘和重排

使用事件委托减少对子元素的操作，事件委托是利用事件冒泡的机制将子元素的操作

利用缓存机制，存储不经常变化的资源来减少对服务器的请求

使用webpack来压缩资源代码体积，删除多余得到代码、注释、简化代码的写法等；使用Tree Shaking删除未使用的代码，使用SplitChunksPlugin进行代码分割（Code Splitting）进行按需加载

LCP 和 FIP

  • LCP 指的事最大内容绘制，常用于衡量页面的加载性能。它测量的是在视口中绘制最大元素所花费的时间，能反映页面主要内容的加载速度，帮助评估用户感知的页面加载完成时间。
  • FIP 指的是首次重要绘制，用于测量将第一个带注释的 HTML 元素绘制到页面上所需的时间。通过对元素进行注释，让创建组件的开发人员无需知道该组件将在哪个页面上使用，它会自动被视为 FIP 候选元素。FIP 旨在更精准地测量页面上对用户而言最重要元素的绘制时间。

懒加载和预加载的区别

预加载通常指的是在用户需要资源之前，自动加载后续可能被使用到的资源，减少后续请求和操作的等待时间，提升用户体验，常用于首屏信息资源的加载，资源的预加载可能同时也会给客户端带来不必要的资源浪费

懒加载通常指的是当资源需要被展示时，才会触发相关加载，能够减少初始页面的加载时间和资源浪费，常用于长页面的资源和图片资源的滚动加载，能够提升页面性能和减少服务器请求消耗，对于图片懒加载常用的方式是添加loading="lazy"属性，对于长页面资源的加载常用的是 Intersection Observer，通过创建观察者的实例对象判断当前元素距离视口的高度来判断资源是否需要被加载，相较于使用滚动事件和防抖来说，前者的性能消耗更小，而且操作更加简单，避免数据的大量计算,但后者的兼容性更强

git

git 和 svn 的区别

git 和 svn 最大的区别在于 git 是分布式的，而 svn 是集中式的。因此不能在离线的状态下使用 svn。如果服务器出现问题，就没有办法使用 svn 来提交代码

svn 中的分支是整个版本库的复制的一份完整目录，而 git 的分支是指针指向某次提交，因此 git 的分支创建更加开销更小并且分支上变化不会影响到其他人。svn 的分支变化会影响到所有人

git 是把内容按元数据方法进行存储，svn 是按文件进行存储

git 的常见命令

配置命令

git config

git init 在当前目录下创建新的 git 仓库

git add <文件名>/. 添加文件到暂存区

git commit -m "提交说明" 提交到本地仓库

git diff 查看工作区和暂存区的差异

git diff --staged 查看暂存区和最新提交的差异

git branch -a 查看所有所有分支

git branch <分支名> 创建分支

git branch -d 删除已合并的分支

git branch -D 强制删除未合并的分支

git checkout <分支名> 或 git switch <分支名> 切换分支

git checkout -b <新分支名> 或 git switch -c <新分支名> 创建并切换分支

git 出现冲突如何解决

冲突产生的原因通常是由于多人对同一文件的同一部分进行了不同的修改，或者在不同分支下对同一文件进行了修改，然后尝试合并修改导致的

解决方法：

使用git status 查看冲突文件，对冲突的文件进行编辑，对冲突的代码区域进行手动修改，修改完成后使用git add 将修改的文件存放到暂存区中，最后使用git commit进行代码提交和合并操作

git 回滚

对于撤销暂存区的修改（使用了git add）

git reset HEAD <file-name>

对于回滚已提交的代码（使用了git commit）

git reset --soft <commit-hash> 撤销提交但保留代码

git reset --hard <commit-hash> 丢弃回滚前的提交和修改

撤销单个提交,生成一个新的提交来撤销之前的更改，可以保留历史记录

git revert <commit-hash>

git pull 和 git fetch 的区别

git fetch 只是将远程仓库的变化下载下来，并没有和本地分支合并。

git pull 会将远程仓库的变化下载下来，并和当前分支合并。

git rebase 和 git merge 的区别

git merge 和git rebase 都是⽤于分支合并，关键在 commit 记录的处理上不同：

git merge 会新建⼀个新的 commit对象，然后两个分支以前的 commit 记录都指向这个新 commit 记录。这种方法会保留之前每个分支的 commit 历史。

git rebase 会先找到两个分⽀的第⼀个共同的 commit 祖先记录，然后将提取当前分⽀这之后的所有 commit 记录，然后将这个 commit 记录添加到⽬标分⽀的最新提交后⾯。经过这个合并后，两个分⽀合并后的 commit 记录就变为了线性的记录了。

webpack

webpack 是一种静态模块打包工具，通过依赖图将前端资源转换成优化的静态资源

常见的配置有：

Entry : 打包入口文件

Output : 打包后的输出目录

Resolve : 用来配置模块的解析方式

Plugin : 扩展功能插件

Module : 用来配置模块如何被解析

webpack 的构建流程

Webpack 的构建流程从初始化开始，读取配置文件与命令行参数，完成编译器（Compiler）对象实例创建；接着从入口文件出发，通过递归解析 import/require 依赖生成依赖图，边解析边触发 Loader 处理各类文件（如用 babel-loader 转译 ES6+ 代码、 css-loader 处理 CSS 依赖），将非 JS 模块转换为 JS 可识别的形式；依赖图构建完成后，使用插件进行优化阶段，通过 TerserWebpackPlugin 压缩代码、 SplitChunksPlugin 提取公共库分包等，并为文件名添加哈希值以避免缓存问题；最终按配置将处理后的模块打包为一个或多个 Chunk 文件（如 main.[contenthash].js），写入到指定输出目录完成构建。

有哪些常见的Loader和Plugin以及两者的区别

Loader 的本质是一个函数，它是一个转换器。webpack 只能解析原生 JS 文件，对于其他类型的文件就需要loader进行转换，Plugin 是一个插件，用于增强 webpack 功能。webpack 在运行的生命周期中会广播出许多事件，Plugin 可以监听这些事件，在合适的时机通过 webpack 提供的 API 改变输出结果

Loader 的配置是在 module.rules 下进行。类型为数组，每一项都是一个 Object，其中包括类型文件 test、加载方式 loader、使用参数 options。Pulign 的配置在 plugins 下。类型也是数组，每一项都是 Plugin 的实例，参数通过构造函数传入

常见的 Loader 和Plugin：

Loader：

image-loader: 加载并压缩图片文件；

css-loader: 加载CSS、处理CSS中的依赖关系；

style-loader: 将CSS代码注入到HTML文档中；

postcss-loader: 自动添加CSS前缀，优化CSS代码等；

vue-loader: 将 .vue 文件编译成 JS 代码；

ts-loader: 使用 ts 编译器将 ts 和 tsx 文件编译成 JS，以及在开发环境中常使用 babel-loader 能够更快的将 jsx 文件进行编译解析。

Plugin:

html-webpack-plugin: 简化HTML文件构建；

webpack-paraller-ugify-plugin: 多进程执行代码压缩，提升构建速度

Webpack 的热更新原理

热更新指的是在不刷新页面的前提下，将新代码替换掉旧代码，HRM（热更新）的原理实际上是webpack-dev-server和浏览器之间维护了一个webscoket服务。当本地资源发生变化后，webpack会先将打包生成新的模块放入内存中，然后webpack-dev-server向浏览器推送更新，并附带上构建时的hash，让客户端和上一次资源进行对比

webpack 的 Tree Shaking 原理

Tree Shaking是一种通过移除多余代码，减少最终生成的代码体积，在生产模式下默认开启，Tree Shaking的基础是 ES6 的模块系统，在 webpack 构建过程中静态分析依赖图，对每个变量、函数、类和导入进行标记来判断是否在实际中被使用，对于未使用的代码在生成最终代码阶段会删除未使用的代码

如何减少 webpack 打包时间

优化构建速度，利用webpack的持久化缓存功能，避免重复构建没有变化的代码，

使用多进程构建，使用happypack等插件将构建过程分解为多个进程，

使用DllPlugin和HardSourceWebpackPlugin实现第三方库的预先打包和缓存中间文件，减少构建时间

如何减小 webpack 打包体积

代码优化，使用 Tree Shaking 删除未使用的代码，

代码分割即 Code Splitting 实现按需加载，

使用压缩工具对静态资源和 JS 代码进行压缩，

将项目中的静态资源使用 CDN 进行加速，减少需要打包的资源

vite 和 webpack 的区别

  • 构建速度不同：webpack 在开发模式下依然会对所有模块进行打包操作，虽然提供了热更新，但大型项目中依然可能出现启动和编译缓慢的问题，Vite利用现代浏览器对ES Module的原生支持，只有在需要时才会编译对应的模块，提升了开发环境的响应速度
  • 打包效率不同：Webpack在打包时，会将所有模块打包成一个或多个bundle文件。这种打包方式虽然简化了生产环境下的资源加载，但也可能导致初次加载速度较慢。Vite在生产环境下，也会将代码打包成一个文件，但它利用了浏览器的缓存机制，只打包和缓存实际改动的模块，提高了打包效率，加快了页面加载速度。
  • 插件生态不同：Webpack拥有庞大的插件生态系统，使得Webpack能够应对各种不同的需求，但也同时也增加了配置的复杂性和学习成本。Vite的插件生态相对较小，对于需求复杂的项目可能需要自定义配置文件来实现定制化功能。
  • 配置复杂度不同：Webpack的配置相对复杂，需要手动配置各种loader和plugin，以及处理不同类型的资源和加载器。Vite通过内置TypeScript、JSX、CSS等现代语法的原生支持，使用esbuild进行预构建依赖，在多数情况下能够开箱即用。

为什么现在仍有大量项目使用 webpack 而不是 vite

`Webpack` 凭借其成熟的生态、丰富的插件体系和对复杂项目的高度可定制性，仍被大量项目尤其是遗留系统和大型应用所采用，而 Vite 虽在开发速度和配置简化上优势显著，但迁移成本、生产环境深度优化需求及特定技术栈的兼容性限制，使得许多团队在权衡稳定性与创新性后，短期内仍倾向于维持现有 `Webpack` 架构。

Eslint 和 Prettier 的区别

Eslint 主要用于代码质量检查，常用于发现和修复语法错误、潜在 bug，提供了代码风格规范和语法检查

Prettier 主要用于代码格式化，确保代码风格统一，但不关注代码逻辑错误
